# AutoSettlement Site Design

## 1. 프로젝트 목적

AutoSettlement MVP의 목적은 정산 담당자가 하나의 정산 작업(batch) 안에서 `라온이앤엠`과 `에스알이앤엠`의 플랫폼별 정산 파일을 함께 업로드하고, 같은 운영 흐름 안에서 파싱 상태를 확인하고, 공통 검수 화면에서 데이터를 점검한 뒤, 회사별 결과 파일을 생성할 수 있도록 사이트 구조를 정의하는 것이다.

본 문서는 업로드된 Stitch 디자인을 화면 설계의 참고 시안으로 해석한 사이트 설계안이다. Stitch 화면을 그대로 구현하는 것이 아니라, 운영자가 실제 업무에서 사용할 수 있는 정보 구조(IA), 페이지 체계, 주요 컴포넌트, 사용자 흐름을 MVP 기준으로 정리한다.

핵심 전제는 다음과 같다.

- MVP의 최상위 작업 단위는 `정산 작업(batch) 1건`이다.
- 하나의 batch 안에서 `라온이앤엠`과 `에스알이앤엠`의 플랫폼별 정산 파일을 함께 업로드한다.
- 검수/오류 확인/출력 흐름은 동일한 화면 맥락에서 처리하되, 데이터와 최종 출력물은 회사별로 분리한다.
- 대상 해상도는 `1920x1080 데스크톱 화면`이다.

## 2. MVP 범위

MVP는 배치 중심의 4단계 운영 구조를 기준으로 설계한다.

1. 플랫폼별 파일 업로드
2. 파싱 및 상태 확인
3. 공통 정산 검수
4. 회사별 출력 파일 생성

MVP에서 반드시 포함되는 업무 조건은 다음과 같다.

- 하나의 batch 안에 `라온이앤엠 업로드 영역`과 `에스알이앤엠 업로드 영역`이 함께 존재한다.
- 회사 전환은 별도 작업 단위가 아니라 `데이터 분리용 탭/필터`로 동작한다.
- 검수는 `공통 검수 화면`에서 수행한다.
- 출력은 회사별로 분리한다.
- 생성 대상 출력물은 회사별 2종이며, 하나의 batch 기준 전체 4개 파일이다.

```text
라온_정산_통합검수용.xlsx
라온_메일러_발송용.xlsx

에스알_정산_통합검수용.xlsx
에스알_메일러_발송용.xlsx
```

MVP에서 다루는 화면은 운영자가 실제 정산 작업을 완료하는 데 필요한 최소 범위로 한정한다. 따라서 업로드, 상태 확인, 오류 확인, 검수, 출력 생성까지는 포함하되, 이메일 발송이나 부가 보정 기능은 포함하지 않는다.

역할 경계:

- AutoSettlement는 플랫폼 원본 파일에서 필요한 값을 추출/계산하고, 공통 `SettlementRow`를 만든 뒤, 회사별 `정산_통합검수용.xlsx`와 `메일러_발송용.xlsx`를 생성하는 사이트다.
- 기존 메일러는 이메일 표시 보정, 작가별 묶기, app 표시/문구 보정, 차감 병합, 실제 발송을 담당한다.
- AutoSettlement는 이메일 본문 렌더링, 메일러 표시 로직 재구현, 작가별 이메일 묶기, 차감 병합, 발송 처리, 이메일러 수정을 하지 않는다.

## 3. 전체 화면 구조

전체 사이트 구조는 `배치 중심 4단계 운영 구조`를 따른다.

```text
정산 작업(batch) 1건
├─ 라온이앤엠 업로드 영역
├─ 에스알이앤엠 업로드 영역
├─ 공통 검수 화면
└─ 회사별 출력
```

Stitch 시안 해석 기준은 다음과 같다.

- 좌측 메뉴는 `단계형 내비게이션`으로 해석한다.
- 회사 전환 UI는 `데이터 분리용 탭/필터`로 해석한다.
- 전체 작업은 `하나의 공통 batch 흐름`으로 해석한다.
- 출력 결과는 `회사별 분리`를 원칙으로 한다.

MVP 기준 전체 IA는 다음과 같다.

```text
AutoSettlement
├─ 배치 목록 / 배치 진입
├─ 배치 업로드
│  ├─ 라온이앤엠 업로드 영역
│  └─ 에스알이앤엠 업로드 영역
├─ 배치 처리 상태
│  └─ 오류 / 누락 / 매칭 실패 상세 섹션
├─ 공통 정산 검수
│  └─ 회사 탭/필터, 플랫폼 필터, 대형 검수 테이블
└─ 출력 생성
   ├─ 라온이앤엠 출력 2종
   └─ 에스알이앤엠 출력 2종
```

오류 상세 화면은 MVP에서 독립 페이지로 분리하지 않는다. `배치 처리 상태 페이지` 또는 `공통 정산 검수 페이지` 내부의 상세 패널/섹션으로 둔다. 이는 IA를 과도하게 무겁게 만들지 않으면서도 Stitch 시안의 흐름과 운영자의 실제 사용 맥락을 유지하기 위한 판단이다.

## 4. 페이지별 설계

### 4.1 배치 목록 / 배치 진입 페이지

목적:
정산 담당자가 생성된 batch 목록을 확인하고, 새 batch를 시작하거나 기존 batch 작업으로 재진입하는 허브 화면이다.

주요 역할:

- 정산월 또는 작업명 기준으로 batch 식별
- 진행 상태 확인
- 오류 발생 여부 확인
- 출력 완료 여부 확인
- 새 batch 생성 진입

주요 컴포넌트:

- 배치 목록 테이블 또는 카드 목록
- 상태 배지(업로드 중, 검수 필요, 출력 가능, 완료 등)
- 오류 건수 요약
- 회사별 출력 생성 여부 요약

MVP 필요성:
필수

### 4.2 배치 업로드 페이지

목적:
하나의 batch 안에서 `라온이앤엠`과 `에스알이앤엠`의 플랫폼별 정산 파일을 함께 업로드하는 핵심 시작 화면이다.

주요 역할:

- 회사별 업로드 영역 분리
- 플랫폼 카드 단위 업로드
- 업로드된 파일 수 및 최근 파일 정보 확인
- 플랫폼별 상태 확인
- 플랫폼별 필수 입력 파일 수 확인

주요 컴포넌트:

- 라온이앤엠 업로드 영역
- 에스알이앤엠 업로드 영역
- 플랫폼 카드 UI
- 파일 상태 요약
- 업로드 진행률 요약
- 브라우저 재진입 시 복원되는 active batch draft state

현재 브라우저 persistence 규칙:

- 현재 MVP shell은 active batch draft를 브라우저 localStorage에서 복원할 수 있어야 한다.
- 복원 대상은 upload metadata / normalized rows / parse issues / selected review row다.
- raw uploaded file binary는 현재 MVP shell persistence 범위에 포함되지 않는다.
- 첫 live upload mutation authority는 `docs/AUTOSETTLEMENT_UPLOAD_MUTATION_CONTRACT.md`를 따른다.
- 현재 live-wired browser upload path는 `sr / misterblue / upload-sr-misterblue`와 `raon / panmurim / upload-raon-panmurim` 단일 XLSX 1-file 경로다.

MVP 필요성:
핵심

### 4.3 배치 처리 상태 페이지

목적:
업로드된 파일이 파싱 가능한 상태인지, 어떤 플랫폼 또는 데이터 항목에서 오류, 누락, 매칭 실패가 발생했는지를 확인하는 운영 점검 화면이다.

주요 역할:

- 배치 전체 상태 요약
- 파싱 단계 진행 상태 확인
- 파일 단위 또는 행 단위 이슈 확인
- 오류 우선순위 파악

주요 컴포넌트:

- 배치 헤더
- 처리 단계 스테퍼
- 상태 KPI 카드
- 오류/누락/매칭 실패 상세 섹션
- 원본 참조 정보 목록

MVP 필요성:
핵심

비고:
오류 상세는 독립 페이지가 아니라 이 페이지의 상세 섹션 또는 패널 구조로 시작한다.

### 4.4 공통 정산 검수 페이지

목적:
라온이앤엠과 에스알이앤엠 데이터를 하나의 batch 맥락에서 함께 검수하되, 회사별 탭/필터로 데이터를 분리해서 검토하는 핵심 검수 화면이다.

주요 역할:

- 전체 정산 결과 검토
- 회사별 필터링
- 플랫폼별 비교
- 이상 항목 재검토
- 정산 금액, 매출, 작품, 작가 단위 확인

주요 컴포넌트:

- 회사 탭/필터
- 플랫폼 필터
- 정렬/검색/필터 바
- 정산 검수용 대형 테이블
- 비고/오류 강조 셀

MVP 필요성:
핵심

### 4.5 출력 생성 페이지

목적:
검수가 끝난 뒤 회사별 산출 파일을 생성하고 다운로드하는 마감 단계 화면이다.

주요 역할:

- 회사별 출력 상태 확인
- 2종 파일 생성 상태 확인
- 생성 일시 및 준비 상태 확인
- 파일 다운로드

주요 컴포넌트:

- 라온이앤엠 출력 카드
- 에스알이앤엠 출력 카드
- 회사별 2종 파일 상태 표시
- 다운로드 액션
- 생성 로그 또는 생성 이력 요약

MVP 필요성:
핵심

출력 정의:

- 회사별 2종 출력
- batch 전체 4개 파일

```text
라온_정산_통합검수용.xlsx
라온_메일러_발송용.xlsx
에스알_정산_통합검수용.xlsx
에스알_메일러_발송용.xlsx
```

## 5. 주요 컴포넌트

### 5.1 Batch 헤더

하나의 batch를 식별하는 공통 상단 영역이다. batch명, 정산월, 현재 단계, 전체 상태, 회사별 출력 준비 상태를 보여준다.

### 5.2 회사 탭 / 데이터 분리 필터

`라온이앤엠`과 `에스알이앤엠` 데이터를 같은 batch 안에서 분리해서 보도록 돕는 UI다. 회사를 작업 자체로 분리하는 것이 아니라, 같은 흐름 안에서 데이터를 전환해 확인하는 목적이다.

### 5.3 플랫폼 카드 UI

플랫폼별 업로드 현황을 카드 단위로 표현하는 컴포넌트다. 플랫폼명, 소속 회사, 업로드 여부, 파일 개수, 최근 파일명, 검증 상태, 오류 여부를 함께 표시한다.

시리즈처럼 하나의 플랫폼 파서가 여러 입력 파일을 하나의 묶음으로 요구하는 경우, 플랫폼 카드는 해당 플랫폼의 필수 파일 수와 현재 업로드된 파일 수를 함께 표시해야 한다. 시리즈 플랫폼은 총 6개 파일을 입력으로 받으며, 일반 매출용 3개와 앱 매출용 3개를 구분해 업로드 상태를 확인할 수 있어야 한다.

문피아 UI 연결 고정 규칙:

- 문피아 카드는 단일 파일 카드로 축약하지 않고 `settlement` / `authorCorrection` 슬롯을 구분해서 보여줘야 한다.
- MVP에서 `settlement`는 required, `authorCorrection`는 optional이다.
- `authorCorrection`는 별도 파일 업로드 슬롯으로만 다루고, UI 직접 입력/인라인 편집 방식은 허용하지 않는다.
- multi-sheet workbook에서 `sheetName`을 UI가 임의 추정하거나 숨기면 안 된다.
- 카드 본문에는 최소한 슬롯명, required/optional 여부, 허용 파일 형식, 업로드 상태, 최근 파일명을 표시해야 한다.
- 현재 mock/UI slice에서는 `BatchPlatformUpload.slots[]`를 통해 슬롯 상태를 표기한다.

네이버/시리즈 샘플 기준 일반 매출용 파일과 앱 매출용 파일은 내부 컬럼 구조만으로 구분되지 않는다. 따라서 업로드 화면은 시리즈 파일을 `일반 매출용 3개`와 `앱 매출용 3개`의 슬롯 또는 파일 그룹으로 받는 구조를 전제로 한다.

### 5.4 처리 단계 스테퍼

`업로드 → 파싱 상태 확인 → 정산 검수 → 엑셀 생성`의 흐름에서 현재 batch가 어느 단계에 있는지 보여주는 진행형 컴포넌트다.

### 5.5 상태 KPI 카드

총 파일 수, 파싱 성공 수, 오류 건수, 매칭 실패 수, 검수 대상 건수 등 운영자가 batch 상태를 빠르게 파악할 수 있는 요약 카드다.

### 5.6 정산 검수용 대형 테이블

MVP의 핵심 컴포넌트다. 작품, 작가, 플랫폼, 회사, 매출, 정산금, 상태, 비고를 한 화면에서 확인할 수 있어야 하며, `1920x1080` 기준에서 고밀도 검수가 가능해야 한다.

### 5.7 오류 / 누락 / 매칭 실패 상세 패널

문제 데이터만 모아서 보여주는 상세 섹션이다. 오류 유형, 원본 행 참조, 누락 필드, 매칭 실패 사유, 영향 범위를 함께 표시한다. 독립 페이지가 아니라 처리 상태 또는 검수 화면 내부의 보조 구조로 둔다.

### 5.8 출력 파일 카드

회사별 결과물을 표현하는 컴포넌트다. 각 회사마다 `정산_통합검수용`과 `메일러_발송용` 파일 상태를 별도 카드 또는 하위 행으로 보여준다.

## 6. 사용자 플로우

MVP 사용자 플로우는 다음과 같다.

1. 운영자가 새 정산 작업(batch)을 생성하거나 기존 batch에 진입한다.
2. 같은 batch 안에서 `라온이앤엠 업로드 영역`과 `에스알이앤엠 업로드 영역`에 플랫폼별 파일을 업로드한다.
3. 업로드 후 시스템은 플랫폼별 파일 상태와 파싱 결과를 표시한다.
4. 운영자는 처리 상태 페이지에서 오류, 누락, 매칭 실패를 우선 확인한다.
5. 운영자는 공통 정산 검수 화면에서 전체 데이터를 회사 필터와 플랫폼 필터를 활용해 검수한다.
6. 검수가 끝나면 출력 생성 페이지로 이동해 회사별 결과 파일 생성을 확인한다.
7. 운영자는 회사별 2종씩, batch 전체 4개 산출물을 다운로드한다.

```text
라온_정산_통합검수용.xlsx
라온_메일러_발송용.xlsx
에스알_정산_통합검수용.xlsx
에스알_메일러_발송용.xlsx
```

이 흐름에서 중요한 점은, 라온이앤엠과 에스알이앤엠이 별도 batch로 쪼개지지 않는다는 점이다. 운영자는 하나의 batch에서 두 회사 자료를 함께 다루고, 검수와 상태 확인은 공통 흐름으로 처리하되, 출력 결과만 회사별로 분리해서 확보한다.

## 7. 데이터 표시 원칙

### 7.1 해상도 기준

기본 설계 기준은 `1920x1080 데스크톱 화면`이다. 주요 검수 정보는 추가 화면 이동 없이 한 화면에서 비교 가능해야 한다.

### 7.2 batch 우선 구조

데이터는 회사별로 분리되지만, 화면 구조는 batch 중심으로 유지한다. 사용자는 “회사 2개를 각각 처리한다”가 아니라 “하나의 정산 작업 안에서 회사 2개의 데이터를 함께 관리한다”는 감각을 가져야 한다.

### 7.3 행 단위 검수 우선

요약 수치보다 실제 정산 행 데이터를 직접 검토할 수 있는 구조를 우선한다. 특히 대형 테이블에서는 행 단위 이상 징후, 금액 차이, 누락 여부를 빠르게 발견할 수 있어야 한다.

### 7.4 오류의 명시적 분리

오류, 누락, 매칭 실패는 정상 데이터와 시각적으로 구분해 표시해야 한다. 단순히 “실패”라고 표시하는 것이 아니라, 왜 문제가 되었는지 사유가 함께 보여야 한다.

### 7.5 회사별 출력 분리

검수 화면은 공통이지만, 출력 정보와 파일명은 반드시 회사별로 분리해 표시해야 한다. 운영자가 잘못된 회사 파일을 다운로드하지 않도록 구분이 명확해야 한다.

### 7.6 범위 밖 액션 비노출

이번 MVP 범위에 없는 이메일 발송, 주소록 보정, 선인세 차감 병합 같은 후속 액션은 버튼이나 메뉴 구조로 노출하지 않는다.

### 7.7 메일러 전달값과 표시 책임 분리

`메일러_발송용.xlsx`는 기존 메일러에 전달할 정산 결과값을 담는 출력물이다. AutoSettlement는 메일러가 사용할 최소 컨텐츠명 전달값을 준비할 수 있지만, 이메일에 표시되는 최종 문구를 렌더링하거나 보정하지 않는다.

`mailerContentTitle`의 역할:

- 메일러에 넘길 최소 컨텐츠명 전달값
- 예: `작품명`, `작품명(app)`, `작품명(이벤트)`, `작품명(이벤트)(app)`
- 실제 이메일 표시 방식은 기존 메일러 책임

시리즈 플랫폼도 같은 원칙을 따른다. 시리즈 파서는 `시리즈 계산방법.xlsx` 기준으로 필요한 결과값만 산출하고, AutoSettlement는 `SettlementRow`와 `메일러_발송용.xlsx`에 필요한 값만 채운다. 이메일에서 어떻게 보일지는 AutoSettlement 화면과 출력 생성 범위에서 처리하지 않는다.

## 8. 제외 범위

이번 문서와 MVP 설계 범위에서 명확히 제외하는 항목은 다음과 같다.

- 이메일 발송 기능
- 이메일 본문 렌더링
- 주소록 보정 기능
- 선인세 차감 병합 기능
- 기존 이메일러 기능의 수정 또는 연동
- 메일러 표시 로직 재구현
- 작가별 이메일 묶기
- 차감 병합 처리
- 파일 파서 구현
- 플랫폼별 파싱 규칙 상세 설계
- 데이터 자동 보정 로직
- 외부 사용자용 권한 체계
- Stitch 시안의 픽셀 단위 재현
- 기능 구현 코드 작성

본 문서는 어디까지나 사이트 구조와 화면 설계안이며, 실제 계산 로직, 파서 로직, 외부 시스템 연동은 다루지 않는다.

## 9. 추후 확장 후보

MVP 이후 확장 후보는 다음과 같다.

- 회사별 검수 화면 완전 분리 모드
- 오류 유형별 전용 대시보드
- 배치 히스토리 비교
- 출력 파일 버전 관리
- 검수 코멘트 및 승인 이력
- 플랫폼별 업로드 템플릿 가이드
- 후속 이메일러 연계를 위한 추가 내보내기 옵션
- 운영 로그 및 감사 이력 화면

단, 위 항목들은 MVP에 포함하지 않으며, 현재 설계에서는 진입점만 암시하지도 않는다. MVP는 배치 중심 4단계 운영 구조를 안정적으로 수행하는 데 집중한다.
