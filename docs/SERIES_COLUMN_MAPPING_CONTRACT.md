# Series Column Mapping Contract

## 1. 목적

이 문서는 `series` 플랫폼의 원본 컬럼 후보와 계산 카테고리 매핑 조사 결과를 문서 계약으로 고정한다.

현재 단계의 목적은 실제 코드 상수, 계산 함수, parser 구현으로 넘어가기 전에 `시리즈 계산방법.xlsx`와 샘플 HTML `.xls`에서 확인한 컬럼 후보를 사람이 검토 가능한 형태로 잠그는 것이다.

이 문서는 구현 문서가 아니다. 코드, 테스트, parser, orchestrator, registry, UI, export, emailer는 수정하지 않는다.

## 2. 확인 파일

조사 기준 파일:

- `시리즈 계산방법.xlsx`
- `contentsSelling_2026-06-08.xls`

두 파일은 시리즈 계산 계약의 근거 자료다.

- `시리즈 계산방법.xlsx`: 카테고리 구분, 설명 텍스트, 요율, 입력 파일 그룹 기준 확인
- `contentsSelling_2026-06-08.xls`: 실제 시리즈 원본 HTML `.xls` 구조와 71개 컬럼 확인

## 3. 계산방법.xlsx 상태

확인 결과:

- sheet: `시리즈`
- 크기: 27행 x 4열
- Excel 수식 셀 없음
- 텍스트 기준의 카테고리 설명과 요율 표가 존재함

의미:

- 계산방법 파일은 Excel formula workbook이라기보다 계산 기준표다.
- 구현은 이 기준표의 텍스트/요율/분류 원칙을 authority로 삼아야 한다.
- 수식 셀을 그대로 실행하는 방식은 현재 기준에서 사용할 수 없다.

## 4. 샘플 HTML .xls 상태

`contentsSelling_2026-06-08.xls` 확인 결과:

- `.xls` 확장자지만 실제 형식은 HTML table
- HTML table 2개 존재
- 실제 데이터는 두 번째 table
- 두 번째 table은 71컬럼 구조
- 마지막 row는 `합계` 행
- 마지막 `합계` 행은 계산 대상 데이터 row에서 제외해야 함

HTML `.xls` adapter 이후 parser가 기대하는 입력:

- 두 번째 table만 추출된 `TabularRow[]`
- header 기반 object
- `sourceFileName`, `sourceRowIndex` 유지
- `합계` 행 제외

## 5. 카테고리 매핑

`시리즈 계산방법.xlsx`에서 확인한 카테고리 매핑은 아래와 같다.

| 정산내역서 구분 | 컨텐츠별 항목 |
| --- | --- |
| 유상-원스토어 | 원스토어 |
| 유상-IAB NPAY | 구글외부쿠키 구매 |
| 유상-IAP | 대여권/소장권 i쿠키, i쿠키로 구매/대여 유상 |
| 유상-웹+자동충전 | 쿠키로 구매/대여, 자동충전 쿠키로 구매/대여 |
| 유상-리딤 | 리딤 |
| 무상이용권 | 모든 무료, 무상 |
| 유상-IAB | 구글로 대여/구매 |

이 매핑은 원본 컬럼명을 직접 코드화하기 전의 문서 계약이다. 실제 코드 상수화는 fixture expected value 검증 후 진행한다.

## 7. 현재 샘플에서 실제 값이 나온 컬럼

`contentsSelling_2026-06-08.xls` 샘플에서 실제 0이 아닌 값이 확인된 주요 계산 후보 컬럼:

- `쿠키로 구매 / 유상`
- `자동충전 쿠키로 구매 / 유상`
- `구글 외부 쿠키로 구매 / 유상`
- `마켓수수료(추정치)`
- `합계`

샘플에서 확인된 보조/식별 컬럼:

- `컨텐츠`
- `컨텐츠No`
- `CP명`
- `작가명`
- `출판사`
- `서비스일`

주의:

- `마켓수수료(추정치)`는 값이 존재하지만 AutoSettlement가 별도 재차감하지 않는다.
- `합계` 컬럼은 원본 검증 참고값으로 볼 수 있으나, 최종 산식은 `시리즈 계산방법.xlsx`와 fixture expected value 기준으로 확정한다.
- 마지막 `합계` 행은 데이터 row에서 제외한다.

## 8. 아직 코드 상수화 금지 원칙

아래 항목은 아직 코드 상수화하지 않는다.

- 원본 컬럼명 목록
- 카테고리별 컬럼 매핑
- `grossSales` 계산식
- `settlementAmount` 계산식
- `합계` 컬럼 검증 방식
- `마켓수수료(추정치)` 처리 방식
- rounding 방식
- 빈값을 0으로 볼지 issue로 볼지에 대한 최종 규칙

이유:

- 현재 샘플에서 실제 값이 나온 컬럼은 71개 중 일부다.
- 전체 6파일 fixture에서 모든 카테고리/slot 조합을 검증해야 한다.
- 바로 코드화하면 누락 또는 오분류 위험이 있다.
- 코드 상수화는 fixture expected value 검증 후 진행한다.

## 9. 다음 단계: fixture expected value 검증 후 코드화

다음 단계 권장 순서:

1. 6파일 fixture manifest 준비
2. HTML `.xls` adapter 결과 기준으로 71컬럼 헤더 확정
3. `합계` 행 제외 확인
4. 카테고리별 후보 컬럼이 모든 fixture에서 존재하는지 확인
5. 각 카테고리별 gross amount expected value 작성
6. 요율 적용 후 settlement amount expected value 작성
7. rounding expected value 확정
8. `sourceRefs` 내부 검증 expected value 작성
9. fixture expected value 검토 완료 후 코드 상수화
10. 그 다음 series 계산 유틸과 group parser 구현
